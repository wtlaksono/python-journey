Prerequisites · Before Day 1
Installing & Configuring Git on macOS
Check, install, verify, and set up Git for the very first time
STEP 01

Check if Git is Already Installed

macOS ships with a system Git bundled inside Xcode Command Line Tools. Before installing anything, check what you already have:

git --version

You'll see one of two outcomes:

OutputMeaningWhat to do
git version 2.x.xGit is installed ✅Skip to Step 4 to verify properly, then Step 5 to configure
command not found: gitGit is not installedContinue to Step 2 or Step 3
A dialog box appears asking to install developer toolsmacOS detected the missing toolClick Install — this runs Step 2 automatically
💡 Note: macOS's system Git is often outdated (version 2.24 or older). It works fine for learning, but installing a fresh version via Homebrew (Step 3) gives you the latest features.
STEP 02

Install Git via Xcode Command Line Tools

This is the simplest method — it installs Git (plus other developer tools) directly from Apple. Run in your terminal:

xcode-select --install

A dialog box will appear on screen. Click Install (not "Get Xcode"). This downloads about 400 MB and takes 3–10 minutes depending on your connection.

Once it finishes, confirm it worked:

git --version
⚠️ Already installed? If you see "xcode-select: error: command line tools are already installed" — that's fine. Git is already there. Run git --version to confirm.
STEP 03

Install Git via Homebrew (Recommended — Gets the Latest Version)

Homebrew is the package manager for macOS. It lets you install and update software from the command line. If you don't have Homebrew yet, install it first:

/bin/bash -c "$(curl -fsSL https://raw.githubusercontent.com/Homebrew/install/HEAD/install.sh)"

Once Homebrew is installed (or if you already have it), install Git:

brew install git

After installation, tell your shell to use the Homebrew Git (not Apple's old one). Add this to your ~/.zshrc:

export PATH="/opt/homebrew/bin:$PATH"   # Apple Silicon
# or
export PATH="/usr/local/bin:$PATH"       # Intel Mac

Then reload your shell:

source ~/.zshrc
💡 Homebrew vs Xcode CLT: Use Homebrew if you want the latest Git (2.4x+), or if you plan to install other tools like uv, node, or python later. It's the professional approach.
STEP 04

Verify Your Git Installation

Run both of these — the first confirms the version, the second shows exactly which Git binary your shell is using:

git --version
which git
CommandGood outputWhat it means
git --versiongit version 2.45.x (or similar)Git is installed and working
which git (Homebrew)/opt/homebrew/bin/gitUsing the Homebrew version ✅
which git (Xcode CLT)/usr/bin/gitUsing Apple's bundled version
💡 Either path works. The Homebrew path just means a newer version. Both are fully functional for everything in this course.
STEP 05

First-Time Configuration — Do This Before Your First Commit

Git needs to know who you are before it can record any commits. These three commands set your identity globally — they apply to every project on your Mac:

git config --global user.name "Your Full Name"
git config --global user.email "you@example.com"
git config --global init.defaultBranch main
SettingWhy it matters
user.nameAttached to every commit you make — visible on GitHub. Use your real name.
user.emailMust match your GitHub account email so commits link to your profile correctly.
init.defaultBranch mainMakes new repos start on main instead of the old default master.
⚠️ Use your real GitHub email. If they don't match, your commits won't show up on your GitHub contribution graph — the green squares won't appear.
STEP 06

Set VS Code as Git's Default Editor

By default Git opens vim when it needs you to write a commit message — which can be confusing for beginners. Set VS Code instead:

git config --global core.editor "code --wait"

The --wait flag tells Git to pause until you close the VS Code tab. Now whenever Git needs a message from you, a familiar editor will open instead of the terminal.

💡 Test it: Run git commit inside any git repo (without -m) and VS Code should open a file called COMMIT_EDITMSG. Write your message, save, and close the tab.
STEP 07

Verify Your Full Configuration

Review everything you've just set up with one command:

git config --global --list

You should see output like this:

user.name=Your Full Name
user.email=you@example.com
init.defaultbranch=main
core.editor=code --wait

You can also check a single value at any time:

git config --global user.email
💡 Made a typo? Just re-run the same git config --global command with the correct value — it overwrites the old one automatically.

git --version

You should see version numbers for both. ✅

💡 Tip: If Cmd + ` doesn't open the terminal on macOS, always use the menu Terminal → New Terminal — it works every time.

STEP 04

Turn On Auto Save

Never lose work again — enable Auto Save:

  1. Press Cmd + Shift + P
  2. Type Auto Save
  3. Click File: Toggle Auto Save

A checkmark appears. Your files now save automatically every time you stop typing.

STEP 05

Set JetBrains Mono as Your Font

Since you have JetBrains Mono installed on your Mac, configure it in VS Code via settings.json:

  1. Press Cmd + Shift + P
  2. Type Open User Settings JSON and hit Enter
  3. Add these lines inside the curly braces:
{
  "editor.fontFamily": "'JetBrains Mono', Menlo, Monaco, monospace",
  "editor.fontSize": 14,
  "editor.fontLigatures": true
}

Save with Cmd + S. The font applies instantly.

✨ What is fontLigatures? JetBrains Mono combines symbols like != -> === into beautiful single glyphs. It makes code much more readable — definitely keep it on!
STEP 06

Change the Color Theme

You'll stare at this editor for hundreds of hours — make it comfortable:

  1. Press Cmd + Shift + P
  2. Type Color Theme
  3. Use arrow keys to preview themes live

Popular choices: Dark+, One Dark Pro, Dracula. Press Enter to confirm.

STEP 07

Split the Editor — Your Superpower 🦸

View two files side by side — incredibly useful when learning:

  1. Right-click hello.py in the Explorer
  2. Click Open to the Side

Now hello.py and day2.py are visible at the same time. Close the split with Cmd + W.

What is Git & Why Does it Exist?
Mental models, core concepts, Git vs GitHub
STEP 01

The Problem Git Solves

Imagine saving an essay like this:

essay.docx
essay_final.docx
essay_final_v2.docx
essay_ACTUALLY_final.docx
essay_ACTUALLY_final_fixed.docx

Sound familiar? Now imagine doing that with thousands of code files and 10 developers editing at the same time. Git solves this by tracking every change to your code — who made it, when, and why.

STEP 02

The Best Mental Model — A Time Machine 🕰️

Think of Git as a time machine for your project folder. Every time you save in Git (called a commit), it takes a snapshot of your entire project:

📸 Snapshot 1 → "Started the project"
📸 Snapshot 2 → "Added login page"
📸 Snapshot 3 → "Fixed the login bug"
📸 Snapshot 4 → "Added dashboard"  ← you are here

At any point you can look back at any snapshot, jump back if something breaks, or see exactly what changed between two snapshots.

STEP 03

The 3 Core Concepts — Learn These Cold

📁
Repository (Repo)
Your project folder that Git is watching. Git secretly creates a hidden .git folder inside it to store all snapshots. You never touch this folder directly.
📸
Commit
One saved snapshot of your project with a message you write. The message explains what you did and why. Good commit messages are a professional skill.
🌿
Branch
A parallel version of your project. You experiment safely on a branch. If it works, merge it back to main. If it breaks, just delete the branch — main stays safe.

Good vs bad commit messages:

❌  "stuff"
❌  "changes"
✅  "Add greeting function to day2.py"
✅  "Fix typo in welcome message"

How branches work:

main branch  (always working, stable code)
─────────────────────────────────────────►
      │
      └──► feature-branch (experiment here)
           ──────────────►
           (if it works → merge back to main)
           (if it breaks → delete it, main is safe)
💡 Think of branches as rough drafts. You write your rough draft separately, and only copy it into the final document when it's ready.
STEP 04

Git vs GitHub — Not the Same Thing

GitGitHub
Tool installed on your computerWebsite in the cloud
Tracks your changes locallyStores a copy online
Works completely offlineLets others collaborate
Like Microsoft WordLike Google Drive

Git is the engine. GitHub is the garage where you park your car so others can see it.

STEP 05

Preview a Real Git History

Let's see what a real project's history looks like. In the terminal:

cd ~
git clone https://github.com/python/cpython.git --depth=5
cd cpython
git log --oneline

You'll see something like:

a3f9d12 Fix typo in documentation
b7c2e45 Add support for new syntax
d1e8f90 Update README

That's thousands of developers, millions of commits, all tracked. Clean up after:

cd ~
rm -rf cpython
💡 Note: Your python-journey folder will show "fatal: not a git repository" if you run git log there — that's normal. We fix it on Day 4!
STEP 06

The Developer Workflow — Preview

Here's the cycle you'll do every single day as a developer. Just read it today — you'll practise it tomorrow:

1. Make changes to your code
        ↓
2. git add         (tell Git which changes to include)
        ↓
3. git commit      (take the snapshot with a message)
        ↓
4. git push        (send snapshot to GitHub)
        ↓
5. Repeat

That's the whole thing. It becomes muscle memory within a week.
