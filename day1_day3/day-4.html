Day 4

Git Basics — Your First Commit

git init git status git add git commit git log git diff
🧠 Quick Recap from Day 3

Today we cover the first two steps of the daily developer cycle. Push comes on Day 6. 

git add      →    git commit    →    git push
(stage it)        (snapshot it)      (Day 6 — not yet)
STEP 01

Initialize Git in Your Project

Open VS Code, open the terminal via Terminal → New Terminal, and navigate to your project:

cd ~/python-journey

Now tell Git to start watching this folder:

git init

You'll see:

Initialized empty Git repository in /Users/yourname/python-journey/.git/

Git is now watching your folder. Confirm it created the hidden .git folder:

ls -la
drwxr-xr-x   .git        ← Git lives here (never touch this)
-rw-r--r--   hello.py
-rw-r--r--   day2.py
💡 The .git folder is Git's brain. It stores every snapshot you ever make. You will never open or edit it manually — Git handles it entirely.
STEP 02

Check the Status — Your New Best Friend

This command tells you exactly what Git sees right now:

git status

You'll see:

On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        day2.py
        hello.py

nothing added to commit but untracked files present
MessageWhat it means
Untracked filesGit can SEE these files but is NOT watching them yet
No commits yetYou haven't taken any snapshots yet
⭐ Golden habit: Run git status constantly — before and after every action. It always tells you exactly where you stand. Senior developers do this dozens of times a day.
STEP 03

Understand the Staging Area

This is the concept most beginners skip — and then get confused later. There are 3 zones in Git:

┌──────────────────────────────────────────────────────────────┐
│                                                              │
│  WORKING DIRECTORY    STAGING AREA        REPOSITORY        │
│  (your files)         (waiting room)      (saved history)   │
│                                                              │
│  hello.py ──git add──► hello.py ──git commit──► 📸 Snapshot │
│  day2.py                                                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
ZoneWhat it is
Working DirectoryYour actual files — where you write code
Staging AreaA waiting room — files chosen for the next snapshot
RepositoryThe saved history of all your snapshots

Think of it like packing a box 📦:

AnalogyGit equivalent
Your room (everything you own)Working Directory
The box (what you're choosing to ship)Staging Area
The delivery truck (sent & saved)Repository (commit history)
CommandAnalogy action
git addPut something in the box
git commitSeal the box and label it
STEP 04

Stage Your Files with git add

Add hello.py to the staging area first:

git add hello.py

Now check the status again:

git status
On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   hello.py

Untracked files:
        day2.py
Status messageMeaning
Changes to be committedhello.py is STAGED — in the box, ready to commit
Untracked filesday2.py is still NOT staged yet

Now add day2.py too — or use the shortcut that adds everything at once:

git add .     # the dot means "add everything in this folder"

Run git status once more — everything should now show under "Changes to be committed".

STEP 05

Make Your First Commit 🎉

Now seal the box and label it. This is your first ever snapshot:

git commit -m "Initial commit: add hello.py and day2.py"

You'll see:

[master (root-commit) a3f9d12] Initial commit: add hello.py and day2.py
 2 files changed, 4 insertions(+)
 create mode 100644 day2.py
 create mode 100644 hello.py

Breaking down the command:

PartWhat it means
git commitTake the snapshot
-mMessage flag — what follows is your description
"Initial commit..."Your description of what this snapshot contains
💡 Congratulations! You just made your first Git commit. Every professional developer remembers this moment. It only gets easier from here.
STEP 06

View Your History with git log

See your snapshot in the time machine:

git log
commit a3f9d12b4c7e8f1d2a3b4c5d6e7f8a9b0c1d2e3f
Author: Your Name <you@example.com>
Date:   Wed Apr 22 15:30:00 2026

    Initial commit: add hello.py and day2.py

That long string of letters and numbers is the commit hash — a unique ID for this snapshot, generated automatically by Git.

For a cleaner view use:

git log --oneline
a3f9d12 Initial commit: add hello.py and day2.py
💡 Tip: Use git log --oneline day-to-day. Use the full git log only when you need to see dates and author details.
STEP 07

Make a Second Commit — Practice the Full Cycle

Open hello.py in VS Code and update it to:

print("Hello! My Python journey starts today.")
print("Day 4: I just learned Git!")

Save it. Now in the terminal check what Git sees:

git status
Changes not staged for commit:
        modified:   hello.py

Git noticed the change. Now do the full cycle:

git add hello.py
git commit -m "Update hello.py with Day 4 progress message"

Check your history — you now have two commits:

git log --oneline
b7c2e45 Update hello.py with Day 4 progress message
a3f9d12 Initial commit: add hello.py and day2.py

You have a real Git history. This is exactly what professional projects look like — just with many more commits.

STEP 08

See Changes Before Staging — git diff

Before staging a file you can see exactly what changed. Add a line to day2.py:

# add this at the bottom of day2.py:
print("Git is starting to make sense!")

Save it, then run:

git diff

You'll see something like:

diff --git a/day2.py b/day2.py
- print("VS Code is my new home!")
+ print("VS Code is my new home!")
+ print("Git is starting to make sense!")
SymbolColourMeaning
+GreenAdded lines
-RedRemoved lines

This tells you exactly what will go into your next commit before you make it. Now commit it:

git add .
git commit -m "Add encouraging message to day2.py"

Run git log --oneline — you should now have three commits total. 🎉

git log --onelineSee compact one-line history
git diffSee what changed before staging
💡 The Golden Rule of Commits

Commit early, commit often. A commit should represent one logical change — not a whole day's work. If your message needs the word "and" a lot, you probably should have made two commits.
